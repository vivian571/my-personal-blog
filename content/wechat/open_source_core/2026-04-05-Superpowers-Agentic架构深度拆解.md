# 🚀 别再盲目写代码了！深拆 superpowers：如何让你的 AI 助手具备“高级架构师”的灵魂？

## 🎯 一个“万星项目”引发的深思：AI 到底缺什么？

不知道你有没有这种感觉：用 Cursor 或者 Claude 写代码，虽然快，但总觉得它像个**“听话但没主见”**的实习生。
你让它加个功能，它哐哐一顿乱改，逻辑跑通了，但代码乱了；你让它修个 Bug，它治标不治本，甚至还引入了新坑。

**AI 缺的不是写代码的能力，而是“开发的脑子”。**

就在最近，GitHub 上那个名为 **obra/superpowers** 的项目瞬间冲破 135k Star，彻底引爆了开发者圈子。它不是又一个基于模型的补丁，它是一套**“Agentic Skills” (代理技能框架)**。

今天，[开源智核] 带你钻进它的底层，看看它如何通过一套严密的“超能力补丁”，把一个只会堆砌代码的 AI，调教成具备“苏格拉底式思维”的架构大牛。

---

## 🧐 大白话拆解：什么是“Superpowers”？

如果把 Claude 比作一只力大无穷但没脑子的**“绿巨人”**，那 superpowers 就是给他穿上了一套带有智能导航和自动平衡系统的**“反浩克装甲”**。

它的核心逻辑本质是 **“方法论的工具化” (Software Development Methodology)**：
1.  **它不让你直接写代码**：当你脑子里只有一个模糊念头时，它会跳出来拦住你，通过“苏格拉底式提问”帮你梳理需求。
2.  **它强迫你做 TDD (测试驱动开发)**：不准先写逻辑再补测试。它会删掉你所有“没经过测试验证”的代码。这叫“RED-GREEN-REFACTOR”的暴力美学。
3.  **它会“分身术”**：把一个大项目拆成无数个 2-5 分钟能搞定的小任务，派出一堆“虚拟小弟 (Subagents)”并行作业。

## 🛠️ 具体详细的操作步骤：如何给你的 AI “换个大脑”？

superpowers 支持多种平台，但最为硬核的还是对 **Claude Code** 的支持。

### 第一步：开启“超能力”实验室
在你的终端里，首先确保装好了 Claude。然后通过以下黑话指令搜索并安装插件：
```bash
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```

### 第二步：注入“超级技能”
安装完成后，在你的项目根目录下启动：
```bash
/add-plugin superpowers
```
你会发现，原本安静的 AI 开始变得“话多”了。当你尝试修改代码时，它会先触发 `brainstorming` 技能。

### 第三步：技能自定义（进阶骚操作）
你可以去修改 `AGENTS.md` 或者 `.claude-plugin` 配置文件。
本质上，每一项“超能力”都是一段被封装的 **Skill 指令**。比如你想让它在提交代码前多加一层“性能审计”，你只需要在 `skills` 目录下克隆一份模板，定义你的 `Audit-Methodology`。

## 💡 核心逻辑：为什么这套“脑子”比代码更贵？（案例分析）

### 案例 A：拒绝“脑热式”开发
**场景**：你对 AI 说“帮我给这个 App 加个注销功能”。
**原生 AI**：直接去改路由，删数据库记录。
**Superpowers AI**：触发 `brainstorming` 技能。它会问：“注销后数据要保留多久？”“关联的第三方授权是否要同步撤回？”“确认删除前是否需要两次校验？”
**本质**：它在用 Socratic Design 帮你规避架构风险。

### 案例 B：强制“洁癖”
**场景**：你想跳过测试，直接上线。
**Superpowers AI**：触发 `test-driven-development` 技能。它会先生成一段失败的测试。如果你想直接写逻辑而不通过这段测试，它检测到 `test baseline` 不匹配，会直接把你的改动回滚。
**本质**：它是一个具有“强制执行力”的流程督导员。

## 🔥 爆款提示词：你可以直接拿走的“Superpowers 骨架”

如果你还没装这个项目，可以尝试把这套“底层本质”输入给你的日常 AI，感受一下“灵魂被洗礼”的感觉：

> **[身份设定]**：你是一位极度冷峻、坚持工程正义的资深软件架构师。
> 
> **[核心本质 - Superpowers 工作法]**：
> 1. **倒带模式 (Step Back)**：禁止直接输出代码。先通过 3 个深度追问，剥离出我需求的底层逻辑，并产出 `DESIGN.md`。
> 2. ** जूनियर 工程师视角 (Junior Engineer Mode)**：请将所有的实施方案拆解为“5分钟内可完成”的极小单元。每个单元必须附带具体的 Git Worktree 分支建议。
> 3. **暴力 TDD**：任何回复中，测试代码必须排在业务代码之前。如果我没有提供测试环境，请先写出环境搭建的命令。
> 4. **真伪校验**：禁止宣称成功。必须提供 `sh` 脚本或单元测试的验证命令，让我亲手看到 `PASS` 标记。
> 
> **[输出格式]**：严格区分 [架构思考]、[行动清单]、[测试先验] 三个模块。

## 🔍 多角度深度分析：这场“代理革命”的终点在哪？

1.  **从效率角度**：**superpowers 解决的是“返工”问题。** 以前 AI 写得快但错得多，现在的 superpowers 让你写得慢一点（因为要思考、要写测试），但总耗时却缩短了，因为你做到了一次性正确（Right First Time）。
2.  **从开发者心智角度**：**这是一种“能力外包”。** 我们把“对工程规范的严苛坚持”交给了 AI。以前我们需要意志力去坚持 TDD，现在，AI 代理成了我们意志力的增量。
3.  **从行业本质看**：**提示词 (Prompt) 正在从“文学创作”转向“控制协议”。** Superpowers 本质上是一套面向 Agent 的“开发协议”。未来，优秀的程序员不需要背诵 API，但需要精通如何编写这样的一套“权力行使准则”。

> **作者建议**：别在生产环境的旧屎山上直接开启 `superpowers` 的 TDD 模式，否则它可能会因为检测到存量 Bug 太多而“崩溃性”回滚你的所有操作。建议从新功能或新分支开始，感受那种“被顶级教练操练”的快感。
